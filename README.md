# Nami 🌊

**BSC（バランスト・スコアカード）に基づく、組織の戦略実行を支える KPI 管理アプリ。**

組織の戦略を 4 つの視点（財務 / 顧客 / 業務プロセス / 学習と成長）で KPI に落とし込み、メンバーがそれぞれ入力・振り返りを行いながらチームの PDCA を回す——その一連の流れを、**「続けられる」こと**を最優先に設計しています。

開発の過程は Zenn「Nami開発日記」で連載中　> https://zenn.dev/ogazon

---

## なぜ作っているか

BSC 型の経営管理ツールはすでにいくつか存在します。機能としては成熟している一方で、**「現場のメンバーが毎月、自分から開いて入力・振り返りを続けられるか」**という観点では、まだ伸びしろがあると感じています。多機能であるほど画面の情報量が増え、「自分が今月何をすべきか」が一目で分からなくなりがちだからです。

Nami は、機能の多さで競うのではなく、**“続く”を生む UX** に焦点を当てます。具体的には、管理者向けの戦略設計画面とは別に、メンバー一人ひとりの「自分の今月」だけが見えるダッシュボードを用意し、入力と振り返りのハードルを下げることを狙っています。

加えてこのプロジェクトは、**フロントエンドからバックエンド・インフラ・IaC・CI/CD・運用までを一気通貫で設計・実装する**ことそのものを目的にしています。各レイヤーで「なぜこの技術・この構成を選んだか」を、開発日記とあわせて記録していきます。

## コンセプト

- BSC の 4 視点で組織の戦略を KPI として管理する
- 各メンバーが自分の担当 KPI を入力し、振り返り（自己評価＋メモ）を残す
- 評価者・上長が振り返りにコメント・評価できる
- 「入力 → 振り返り → 改善」のループを**続けやすい UX**で支え、組織と個人の成長を後押しする

### 設計判断の記録（なぜこの形にしたか）

技術選定や仕様は「定番だから」ではなく、このプロダクトの目的に対して選んでいます。主な判断は以下のとおりです。

- **主語は「組織」**：BSC は本来、組織の戦略管理フレームワーク。まずは正統な「組織の BSC」として設計し、個人最適化された発展形は将来の差別化テーマとして温存する
- **MVP では 4 視点を固定**：視点の追加・階層化は魅力的だが構造設計が重くなるため後回しにし、まず普通の BSC アプリを確実に動かす
- **権限はロールベースで段階導入**：MVP は管理者ロールのみで動かし、データモデルには `role` / `company_id` / `department_id` を最初から予約。評価者・一般メンバーの権限制御は後続バージョンで追加する（「管理者＝全権を持つメンバーのスーパーセット」として設計）
- **KPI の極性（polarity）を持つ**：売上のように「上昇が good」な指標と、コストのように「下降が good」な指標で達成判定の向きが逆になるため、指標ごとに極性を保持して判定を自動化する
- **戦略マップ（因果関係の可視化）は専用バージョンで集中実装**：他画面はマップに依存しないため、データ上の「KPI ↔ 戦略目標の紐づけ」は MVP に含めつつ、GUI でのマップ描画は独立したバージョンで作り込む

## ロードマップ（段階リリース方針）

**「薄くても動くものを公開し、バージョンを重ねて育てる」**方針です。各バージョンは必ずクラウド上にデプロイし、インターネットから動作確認できる状態を 1 区切りとします。

| バージョン | 内容 |
|---|---|
| v0.1–0.2 | ローカル土台 → クラウド最小公開 → DB 接続 |
| v0.3 | 認証（ログイン） |
| v0.4 | 戦略目標の管理 |
| v0.5 | KPI 管理（戦略目標への紐づけ） |
| v0.6 | 当月実績入力（極性に基づく達成判定） |
| v0.7 | 行動計画＋振り返り（自己評価・証憑アップロード） |
| v0.8 | マイダッシュボード（“続く”の核） |
| **v1.0** | **UI を磨き込み、実用に耐えるレベルへ** |
| v1.5 | 戦略マップ（因果関係の GUI 可視化） |
| v1.6 | ロール制御（評価者・一般メンバー） |
| v1.7 | スコアカードのフルビュー |
| v2.0 | ゲーミフィケーション（貢献度の可視化・表彰） |

## 技術構成

| レイヤー | 技術 | 選定理由 |
|---|---|---|
| フロントエンド | Next.js | モダンフロントの標準。あえてマネージドホスティングではなく AWS に載せ、AWS 全体を学ぶことを優先 |
| バックエンド | Laravel | リレーション・集計が多い要件にマッチし、開発効率も高い |
| DB | Aurora Serverless v2 (MySQL) | 集計・前年同月比・リレーションが多く RDB が最適。Serverless v2 は未使用時にスケールダウンでき、コストを抑えられる |
| 認証 | Amazon Cognito | 認証基盤を自前実装せずマネージドに寄せる |
| コンテナ | Amazon ECS Fargate | 常駐する Web アプリ向け。EC2 を自前管理しないマネージドコンテナ構成 |
| CI/CD | GitHub Actions → ECR → ECS | デプロイを自動化し、バージョン更新のたびに繰り返し実践する |
| IaC | Terraform | インフラをコード管理。`terraform destroy` で一括削除でき、消し忘れ課金を防ぐ |
| 開発環境 | Docker Compose | ローカルで MySQL・Laravel・Next.js を完結させ、開発中のクラウド課金をゼロにする |

### アーキテクチャの設計思想：サーバレス vs 常駐の使い分け

本構成は「完全サーバレス」ではなく、**マネージドコンテナ（Fargate）を主軸に、一部をサーバレスで補うハイブリッド**です。

- **常駐が向く処理 → ECS Fargate**：常時アクセスを受ける Web アプリ本体（Next.js / Laravel）
- **イベント駆動が向く処理 → Lambda**：たまに動く処理（将来の集計バッチ・通知など）
- **完全マネージド → Cognito / S3**：自前管理不要

「EC2（生のサーバ）は管理しないが、Web アプリは常駐させる」という選択です。常駐型のため立てっぱなしは課金が続く点に注意し、開発中はローカル中心・クラウドは検証時のみ起動する運用とします。

## 費用方針 💰

**鉄則：開発はローカル中心。クラウドは「試す時だけ立てて、Terraform で壊す」。**

| フェーズ | 内容 | 費用感 |
|---|---|---|
| ① 開発中 | ローカル Docker（MySQL/Laravel/Next.js 全部ローカル） | **ほぼ 0 円**。開発時間の大半はここ |
| ② クラウド検証 | デプロイして動作確認、終わったら `terraform destroy` | 起動した時間分のみ。数百円〜のレンジ |
| ③ 常時稼働 | 実運用するフェーズ | 月数千円〜（構成・稼働時間による） |

### コストを抑える具体策

- **Aurora Serverless v2 の最小 ACU を 0 に設定可能**（2024/11 のアップデート）。アクセスがない間は DB が自動一時停止し、コンピューティング料金は発生しない（ストレージ料金のみ）。※再開に最大 15 秒程度かかる点は開発では許容
- **東京リージョンの Aurora Serverless v2 は 2024/11 に 25% 値下げ済み**
- **NAT Gateway は立てっぱなしだと地味に高い**。検証が終わったら必ず削除
- **AWS Budgets で予算アラートを最初に設定**（例：月 1,000 円超でメール通知）
- **Terraform で全リソース管理 → `terraform destroy` で一括削除**し、消し忘れ課金を防ぐ

> 正確な金額は変動するため、本番前に必ず [AWS Pricing Calculator](https://calculator.aws/) で試算してください。

## 開発フロー

```
1. ローカル Docker で各バージョンを作る（0円）
        ↓
2. Terraform で AWS 環境を定義
        ↓
3. デプロイしてインターネット上で動作確認 → 確認後 terraform destroy
        ↓
4. バージョンを上げるたびに 2〜3 を繰り返す（CI/CD で自動化）
```
